## 思路（必填）
### 问题分析
课程安排图是否是 **有向无环图(DAG)**。这个题目本质上是一个有向图的环检测问题。每门课程是一个节点，先修关系是一条有向边。如果图中存在环，意味着课程之间存在循环依赖，永远无法修完；如果无环，则一定可以按顺序修完。
### 算法选择
选择了**拓扑排序（BFS 实现的 Kahn 算法）**。这种算法通过不断剥离图中入度为 0（没有前置依赖）的节点来推进，非常符合“修课”的物理直觉。
### 实现步骤
1. **建图与统计入度**：遍历 `prerequisites`，统计每个节点的入度（依赖数），生成 入度表 `indegrees`，并构建邻接表（每门课学完后能解锁哪些课）。
2. **初始化队列**：借助一个队列 `queue`，将所有入度为 0 的节点（可以直接修的课）入队。
3. **BFS 遍历（剥洋葱）**：当 `queue` 非空时，依次将队首节点出队，在课程安排图中删除此节点 `pre`：
	- 并不是真正从邻接表中删除此节点 `pre`，而是将此节点对应所有邻接节点 `cur` 的入度 −1，即 `indegrees[cur] -= 1`。
	- 当入度 −1后邻接节点 `cur` 的入度为 0，说明 cur 所有的前驱节点已经被 “删除”，此时将 `cur` 入队。
4. **环检测判定**：在每次 `pre` 出队时，执行 `numCourses--`；
	1. 若整个课程安排图是有向无环图（即可以安排），则所有节点一定都入队并出队过，即完成拓扑排序。换个角度说，若课程安排图中存在环，一定有节点的入度始终不为 0。
	2. 因此，拓扑排序出队次数等于课程个数，返回 `numCourses == 0` 判断课程是否可以成功安排。



## 关键点
>[!Tip] 
>入度数组 `indegress` 的状态机维护，以及巧用 `numCourses--` 来作为判断最终结果的指标。

## 代码

```java
class Solution {  
    public boolean canFinish(int numCourses, int[][] prerequisites) {  
        int[] indegress = new int[numCourses];  
        List<List<Integer>> adjacency = new ArrayList<>();  
        Queue<Integer> queue = new LinkedList<>();  
        for (int i = 0; i < numCourses; i++) {  
            adjacency.add(new ArrayList<>());  
        }  
  
        for (int[] cp : prerequisites) {  
            indegress[cp[0]]++;  
            // 打开第 cp[1] 个小抽屉, 把cp[0]放进去  
            adjacency.get(cp[1]).add(cp[0]);  
        }  
  
        for (int i = 0; i < numCourses; i++) {  
            if (indegress[i] == 0) {  
                queue.add(i);  
            }  
        }  
  
        while (!queue.isEmpty()) {  
            // 取出一门可以直接修的课  
            int pre = queue.poll();  
            // 剩余需要修的课程总数减 1            
            numCourses--;  
            // 遍历这门课解锁的所有后续课程  
            for (int cur : adjacency.get(pre)) {  
                // 把后续课程的入度减 1（消除一个前置依赖）  
                if (--indegress[cur] == 0) {  
                    // 如果减完后，这门后续课的入度也变成 0 了，说明可以修了，入队  
                    queue.add(cur);  
                }  
            }  
        }  
        return numCourses == 0;  
    }  
}
```

## 知识点
### 1. 核心概念与关键变量

在理解代码前，我们需要将课程关系映射为“图”的概念：

- **节点（Node）：** 每门课程就是一个节点。
- **有向边（Edge）：** 如果修课程 A 必须先修课程 B（即 `[A, B]`），则存在一条从 B 指向 A 的边 `B -> A`。
- **入度（In-degree）：** 指向某个节点的边的数量。在问题中，**入度就是一门课程还缺少的前置课程数量**。入度为 0，意味着这门课可以直接上。

 代码中定义了三个核心数据结构：

- `int[] indegress`：**入度数组**。`indegress[i]` 存储课程 `i` 的前置课程数量。
- `List<List<Integer>> adjacency`：**邻接表**。`adjacency.get(i)` 存储了“修完课程 `i` 后，能直接解锁的后续课程列表”。
- `Queue<Integer> queue`：**队列**。专门用来存放当前**入度为 0**（没有任何前置要求，或者前置要求已全部满足）可以立即去修的课程。

### 2. 邻接表（Adjacency List）

```java
for (int[] cp : prerequisites) {  
    indegress[cp[0]]++;  
    // 打开第 cp[1] 个小抽屉, 把cp[0]放进去  
    adjacency.get(cp[1]).add(cp[0]);  
}
```
#### 1. `get(cp[1])` 的字面意思

在代码初始化阶段，`adjacency` 被定义为一个 `List<List<Integer>>`。你可以把它想象成一个**大抽屉**，里面有许多个**小抽屉**：

- **`adjacency`（大抽屉）**：代表整张图，索引 `0` 到 `numCourses - 1` 对应每一门课。
- **`adjacency.get(i)`**：就是打开第 `i` 个小抽屉。
- **`cp[1]`**：在课程对 `[1, 0]` 中，`cp[1]` 是 `0`（先修课）。

所以，`adjacency.get(cp[1]).add(cp[0])` 的意思就是：**“打开先修课（`cp[1]`）的抽屉，把想修的后续课（`cp[0]`）放进去。”**

#### 2. 循环完的 `adjacency` 是什么样子？

循环结束后的 `adjacency` 记录了 **“每门课学完后，能直接解锁哪些课”**。

##### 举个具体的例子：

假设有 4 门课（0, 1, 2, 3），依赖关系如下：

1. `[1, 0]`：学 1 前先学 0
2. `[2, 0]`：学 2 前先学 0
3. `[3, 1]`：学 3 前先学 1
4. `[3, 2]`：学 3 前先学 2

##### 循环后的 `adjacency` 内部结构：

|**索引 (先修课)**|**存储的列表 (被解锁的课)**|**物理意义**|
|---|---|---|
|**0**|`[1, 2]`|修完 0，可以去修 1 和 2|
|**1**|`[3]`|修完 1，可以去修 3|
|**2**|`[3]`|修完 2，可以去修 3|
|**3**|`[]`|修完 3，没有后续课程了|
### 3.邻接表 和 入度
**邻接表**和**入度**，本质上是站在**一条边的两端**，向着完全相反的方向看问题。
可以把有向边 `A -> B`（修完 A 才能修 B）拆解成这两种截然不同的视角：
#### 1. 邻接表：站在起点的“向外看”（出边视角）

邻接表 `adjacency` 的视角是在**前置课程（A）** 上，它关心的是 **“未来”**：
- **内心活动：** “我（A）修完之后，能顺带解锁哪些课（B, C, D...）？”
- **作用：** 它是 **“钥匙串”**。当你完成了一门课，邻接表负责告诉你：“拿着，去把这几门课的进度往前推一步。”

#### 2. 入度数组：站在终点的“向内看”（入边视角）

入度数组 `indegress` 的视角是在**目标课程（B）** 上，它关心的是 **“过去”**：
- **内心活动：** “要想修我（B），前面还有几个大爷（A, E, F...）拦着我？”
- **作用：** 它是 **“门上的锁”**。入度是 3，说明门上挂了 3 把锁；入度是 0，说明门上没有锁，你可以直接推门进去（入队）。
#### 3. 完美的配合：锁与钥匙的接力赛

为什么这两者必须“相反”且配合使用？因为拓扑排序本质上是一个**不断开锁**的过程。

1. **找没锁的门：** 一开始，我们看 `indegress`，找出所有入度为 0（没有锁）的课，放进队列。
2. **进门拿钥匙：** 从队列里拿出一门课（比如 A）修完。此时去看 A 的 `adjacency`。
3. **去开后面的锁：** `adjacency` 告诉你 A 能解锁 B 和 C。于是你跑到 B 和 C 的门前，把它们的 `indegress`（锁的数量）减去 1。
4. **门开了：** 如果 B 的 `indegress` 减到了 0，说明拦着 B 的最后一门课也被你修完了，B 这扇门彻底开了，把 B 扔进队列。
#### 4. 违和感
我们记录依赖关系时，给的数据通常是 `[目标课, 前置课]`（一种向回看的倒推关系），但我们在执行搜索时，必须是**顺着时间线往前走**的（修完前置，再去解锁目标）。

为了把这两种方向统一起来，算法刻意设计了这套“相反”的结构：

- 用 **入度（向后看）** 来卡住进度，判断能不能修。
- 用 **邻接表（向前看）** 来推动进度，顺藤摸瓜找下一门课。

### 4. BFS 核心遍历（修课过程）

```java
while (!queue.isEmpty()) {  
    // 取出一门可以直接修的课  
    int pre = queue.poll();  
    // 剩余需要修的课程总数减 1    
    numCourses--;  
    // 遍历这门课解锁的所有后续课程  
    for (int cur : adjacency.get(pre)) {  
        // 把后续课程的入度减 1（消除一个前置依赖）  
        if (--indegress[cur] == 0) {  
            // 如果减完后，这门后续课的入度也变成 0 了，说明可以修了，入队  
            queue.add(cur);  
        }  
    }  
}
```
这是最巧妙的一步：

1. 每次从队列拿出一门没有前置依赖的课 `pre`，这代表我们“修完”了这门课。
2. 利用 `numCourses--` 作为计数器。每修完一门，剩余课程数减 1。
3. 去邻接表里找到修完 `pre` 能解锁的后续课程 `cur`。
4. 将这些后续课程的入度减 1（`--indegress[cur]`）。如果减到了 0，说明它的前置全被满足了，扔进队列里等待下次被修。


## 复杂度

- Time: O(**N+M**),遍历一个图需要访问所有节点和所有临边，N 和 M 分别为节点数量和临边数量；
- Space: O(**N+M**),为建立邻接表所需额外空间，`adjacency` 长度为 N ，并存储 M 条临边的数据。
